# GENESIS 2.5 — Experimental Setup and Reproduction

This folder is the self-contained reproducibility package for the GENESIS 2.5
accelerator benchmarks (paper: `paper/manuscript_softwarex_draft.tex`). It
records the exact hardware, software versions, build recipes, and commands used,
so every number in the paper can be regenerated.

Two platforms were used:
- **Laptop (OpenCL)** — all published OpenCL speedup/scaling/divergence data.
- **RunPod RTX 4090 (CUDA)** — CUDA-backend correctness + speedup validation.

---

## 1. Hardware

| | Laptop (OpenCL arm) | RunPod pod (CUDA arm) |
|---|---|---|
| GPU | AMD Radeon 890M (RDNA 3.5, integrated) | NVIDIA GeForce RTX 4090 (Ada, `sm_89`, 128 SMs) |
| GPU driver / stack | Mesa **rusticl** (OpenCL 1.2, fp32 only, no `cl_khr_fp64`) | NVIDIA **570.195.03**, CUDA 12.8 |
| VRAM | shared (integrated) | 24 GB |
| GPU max clock | — | 3105 MHz (SM); benchmarked with clocks locked |
| CPU | AMD (AVX-512, Strix-class APU) | AMD Ryzen 9 9950X (16C/32T) |
| System RAM | 78 GB | 123 GB |
| OS | Linux 6.17 | Ubuntu 24.04.3 LTS |

The two arms are compared **only within a platform** (CPU vs GPU on the same
machine); laptop and pod numbers are not cross-compared.

## 2. Software

| Component | Laptop | Pod |
|---|---|---|
| Compiler | GCC (system) | GCC 13.3.0 |
| CUDA toolkit | — | nvcc 12.8.93 (`/usr/local/cuda-12.8`) |
| OpenCL loader | system `libOpenCL.so` | `ocl-icd-opencl-dev` (installed; loader stub only) |
| GENESIS build | `nxgenesis` (headless, no X11) | same |
| Analysis | Python 3, matplotlib | Python 3 |

Container image (pod): `runpod/pytorch:...-cuda12.8...-ubuntu2404` (the CUDA
toolkit ships in `/usr/local/cuda-12.8`; the image lacks an OpenCL loader, which
must be installed — see build gotchas).

## 3. Binaries and the two-arm design

Both arms are the **same headless `nxgenesis` family** (no X11 → avoids the
GUI-linked-binary confound documented in the paper). They differ only in the
accelerator backend:

| Binary | Backend | Precision | Build flag |
|---|---|---|---|
| `nxgenesis_nocl` | none (CPU) | fp64 | — |
| `nxgenesis` (laptop) | OpenCL | fp32 kernel | `USE_OPENCL=1` |
| `nxgenesis` (pod) | CUDA | fp32 kernel | `USE_CUDA=1` |

GPU dispatch is confirmed at run time, never assumed: each accelerated run emits
a device banner + dispatch report (`OCL MULTILOOP …` / `CUDA MULTILOOP …`).

## 4. Build recipe (verified)

GENESIS's ancient recursive Makefiles need a few one-time steps on a clean clone.
The full sequence is in `paper/REPLICATION.md` and automated in
`genesis/src/hines/cuda/runpod_validate.sh`. Summary:

1. **netcdf/diskio libs** (needed because the bundled netcdf-3.4 `configure`
   false-detects cross-compiling): `macros.make` + `ncconfig.h` are committed, so
   build `libnetcdf.a`, `netcdflib.o`, `FMT1lib.o` explicitly, then `make nxgenesis`.
2. **CUDA backend** (`genesis/src/hines/cuda/`): `make USE_CUDA=1` with
   `nvcc -arch=sm_89 -ccbin gcc`, then link `nxgenesis` with `-lcudart`.

### Build gotchas found on the RunPod pod (real-hardware validation)
These were caught by actually building on a fresh pod, not assumed:

| Symptom | Cause | Fix |
|---|---|---|
| `macros.make: No such file` | netcdf configure aborts (cross-compile false positive) | `macros.make` committed (07cbc42) |
| `No rule to make FMT1lib.o` | diskio format lib not auto-built | build it explicitly (REPLICATION Step 1) |
| `cannot find -lOpenCL` (blocks even `code_g`) | image has CUDA but no OpenCL loader | `apt-get install -y ocl-icd-opencl-dev` |
| `No rule to make sim/simlib.o` | cascade from the `-lOpenCL` failure above (code generator never links) | fixed once `libOpenCL.so` present |
| `undefined reference to tgetstr` | CUDA link dropped termcap libs via `EXTRALIBS` override | link with explicit `-lncurses -ltinfo … -lcudart` order |
| neurons frozen at `initVm` (false "agree") | benchmark `include` used a hardcoded absolute path to `genesis/startup/schedule.g` (untracked install dir) | use the tracked path `genesis/src/startup/schedule.g` |

## 5. Experiments

### 5.1 OpenCL campaign (laptop) — `run_overnight_campaign.py`
Four studies, wall-clock, both arms identically instrumented:
- **Study 1** — CPU-vs-GPU scaling at K=50000 (headline speedup table).
- **Study 2** — speedup vs simulation length K (amortization of model build).
- **Study 3** — fp32-vs-fp64 numerical divergence (correctness).
- **Study 4** — long-run reproducibility.

Data: `data/campaign_wallclock_summary.csv`, `campaign_ksweep.csv`,
`campaign_divergence.csv`, `campaign_reproducibility.csv`. Figures:
`paper/figures/fig_campaign_*.png` (generated by `plot_campaign.py`).

### 5.2 CUDA validation (RunPod RTX 4090) — `runpod_validate.sh`
- **Correctness**: `hh1952_ap_verify.g`, 8 HH neurons, 2 nA step, 200 steps.
  CPU fp64 vs CUDA fp32 multiloop.
- **Speedup**: `hh_spiking_benchmark.g`, K=50000, N sweep, clocks locked.

Results: `cuda_validation/` (see that folder's README).

## 6. Headline results

- **OpenCL (Radeon 890M)**: step-phase speedup peaks ~18× (sustained) / ~13.5×
  (cold-start) near N=5000; ~1.5×10⁹ neuron-steps/s; end-to-end up to ~4×.
- **CUDA (RTX 4090)**: backend compiles on nvcc 12.8, dispatches on the GPU, and
  matches fp64 CPU to **7×10⁻⁸ V** over an action potential
  (`RESULT_VM`: CPU −0.02143485882, CUDA −0.02143492922; `NEURONS_AGREE: YES`).
  Speedup sweep: see `cuda_validation/`.
- **fp32 vs fp64**: agreement to ~2×10⁻⁷ V over one AP on both backends; spike-
  phase drift over long runs, as expected for reduced-precision oscillator
  integration.

## 7. One-command reproduction

- Laptop OpenCL campaign: `python3 experiments/run_overnight_campaign.py`
- Pod CUDA validation: `bash genesis/src/hines/cuda/runpod_validate.sh`
  (on an `sm_89` CUDA pod; installs `ocl-icd-opencl-dev` and builds both arms).

Exact commit for the paper: see `paper/manuscript_softwarex_draft.tex` field C1.
