Key findings (this project)
Technical record
F1
Scheduler bottleneck is the dominant overhead. GENESIS's simulation loop iterates over all absorbed elements every step regardless of GPU offload. At N ≥ 2000, this accounts for 98–99% of the GPU arm's total step time (~25–45 ns/element/step). End-to-end speedup is capped at 1.1–1.4× without restructuring the scheduler.
F2
After scheduler fix + fp32 port, multiloop mode achieves 3.2–16.2× end-to-end speedup. Raw kernel speedup is 4–149×. The per-step dispatch path remains slower than CPU due to per-step upload overhead dominating the small kernel cost.
F3
Three bugs in hines_init.c / hines_solve.c silently broke Vm for all multi-neuron hsolve architectures. SKIP_DIAG OOB read, missing parentno==-1 guard in backward pass, and wrong ncompts==1 proxy. Classic GENESIS single-neuron usage never triggered any of them.
F4
AMD Radeon 890M (RDNA3 iGPU) has zero fp64 support. Any code path passing double-precision data to the GPU silently falls back to CPU. The kernel must run in float; double↔float conversion happens at the host boundary in ocl_hsolve.c.
F5
All three execution paths agree to fp32 precision on single-step dynamics. CPU no-OCL, GPU per-step (chip_channel_update), and GPU multiloop (chip_channel_multiloop) give Vm within 3×10⁻⁹ V after one 50 µs step with active HH channels and 2 nA injection, for N=1 and N=2.
F6
OCL platform path matters critically. OCL_ICD_VENDORS=/etc/OpenCL/vendors_rocm (old path, non-existent) causes silent CPU fallback — the binary reports no error, runs on CPU, produces misleading "GPU" results. Correct path: /run/host/etc/OpenCL/vendors.
